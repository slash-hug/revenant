# Revenant v1.1 — Hardening Design

- **Date:** 2026-06-14
- **Status:** Approved (brainstorm) — ready for `feature-research`
- **Author:** Randy + Claude
- **Scope class:** Tier-2 (multi-file, cross-workstream, architectural decision in WS-1)

## Motivation

A consolidated review (design critique 36/40, a competitive/market study, and code-quality + performance audits) surfaced a set of correctness, data-integrity, and cheap performance gaps. This round is **hardening only**: make what the product already advertises actually work, correctly and fast. Net-new features, larger refactors, and product/positioning work are deliberately deferred to tracked GitHub issues (#1–#14).

The headline finding: the product's core promise — *annotations survive document edits via fuzzy re-anchoring* — does not work today. `reanchor.rs` has zero callers, and annotation context (the surrounding lines the matcher needs) is dropped at the IPC seam, so it could not work even if wired. This is also the differentiator the market research named as Revenant's durable moat. WS-1 fixes it.

## Scope

**In scope** (this spec):
- WS-1: Re-anchoring made real (all code-audit P0-1, P2-1).
- WS-2: Data-integrity correctness (P0-2, P1-1, P1-2, P1-3, P1-4, P1-5).
- WS-3: Cheap perf/stability wins (highlight.js curate, Mermaid trim, WebGL dispose leak).

**Out of scope** → tracked issues (do not implement here):
- Perf refactors: tab-switch persistence (#1), preview render-guard/`afterUpdate`→`$effect` (#2), scroll-sync (#3), obsidian async (#4), watcher debounce (#5).
- Features: PDF/HTML export (#6), KaTeX (#7), Linux (#8).
- UX: ⌘K palette (#9), keyboard add-comment (#10), undoable delete + status-bar dirty (#11).
- Code-quality/security: confinement fail-open + DOMPurify allowlist (#12), settings path + schema-policy unification + P3 cleanups (#13).
- Product: positioning & pricing (#14).

## Decisions (locked in brainstorm)

1. **Re-anchoring — Approach A: Rust computes context, IPC contract stays frozen.** Context lines are re-anchoring fuel, internal to the Rust matcher; the frontend never reads them. Rust derives them from the canonical doc at save time. This respects the frozen contract (whose mismatch with the richer on-disk schema *caused* the original bug), centralizes the logic next to `reanchor.rs`, deletes dead frontend code, and sidesteps the preview-anchor `end_char` bug (P2-1) by working at line granularity. Cost: one extra doc read on save (negligible; we already hash on save).
2. **`save_file` concurrency — advisory exclusive file lock (new dependency).** Make the optimistic-concurrency read-hash-compare-write atomic. Preferred crate: `fd-lock` (maintained, cross-platform advisory locks); `fs2` acceptable fallback.
3. **CRLF — CRLF-aware parse, preserve line endings.** Make the frontmatter parser + byte-offset math handle `\r?\n` and never rewrite the file's original endings (no surprise Windows diffs).

## WS-1 · Re-anchoring made real

**Files:** `src-tauri/src/ipc.rs`, `src-tauri/src/reanchor.rs`, `src-tauri/src/annotations.rs`, `src/lib/EditorPane.svelte`, `src-tauri/src/tests/`.

### Changes
1. **`ipc.rs` `save_annotations`** — before persisting, read the doc at `doc_path` and, for each annotation, derive `context_before` / `context_after` from its line range (the line immediately before `line_start` and the line immediately after `line_end`, matching the matcher's `context_before + quoted_text + context_after` model). Store them in the sidecar via `ann_from_ipc`. **Stop hard-coding them to `String::new()`** (`ipc.rs:163-164`). The IPC `Annotation` type is unchanged.
2. **`ipc.rs` `load_annotations`** — after loading the sidecar, compare `sidecar.doc_content_hash` to the freshly-read doc hash (already computed at `ipc.rs:466-470`). If they differ, run `reanchor::reanchor_all` against the fresh doc, update each annotation's line range + `status` (`anchored` / `detached`), and persist the re-anchored sidecar (so the new hash + positions stick). Return the updated set.
3. **`reanchor.rs` — bound the algorithm** (it is O(annotations × doc × chars) today and would freeze the UI once wired):
   - Keep the content-hash short-circuit and exact verbatim probe first; **skip fuzzy entirely on a verbatim hit**.
   - Restrict the fuzzy search to a window of **±W lines** (default `W = 50`, named constant) around the stored `line_start`, not the whole doc.
   - Pre-filter candidate windows with a cheap line length/token ratio before running the char-level `TextDiff`.
   - If no window scores ≥ 0.75, mark `detached`.
   - Run re-anchoring off the command thread (`tauri::async_runtime::spawn_blocking`) when the doc is large, so `load_annotations` never blocks the UI.
4. **`EditorPane.svelte`** — delete the now-dead `context_before` / `context_after` computation in `handleSelectionChange` (it was discarded at the IPC seam; Rust owns it now).

### Acceptance
- Open a doc, add an annotation, edit the doc (insert lines above the anchor), reload → the annotation re-anchors to the moved text (line numbers updated, `status: anchored`).
- Delete the anchored text, reload → `status: detached`, not silently mis-positioned.
- Re-anchoring 100 annotations on a 5000-line doc completes without a perceptible UI stall.

### Tests
- **New integration test** (`src-tauri/src/tests/`): the save→edit→load→re-anchor round-trip end-to-end (the cross-module gap the 73 unit tests missed).
- Unit tests for the windowed search + verbatim short-circuit + detach threshold.

## WS-2 · Data-integrity correctness

**Files:** `src-tauri/src/file_io.rs`, `src-tauri/src/annotations.rs`, `src-tauri/src/frontmatter.rs`, `src/lib/stores/annotations.ts`, `src/lib/EditorPane.svelte`, `src-tauri/Cargo.toml`.

### Changes
1. **`save_file` advisory lock** (`file_io.rs:101-124`) — open the target once, take an exclusive advisory lock (`fd-lock`), re-hash from that handle, compare to `expected_hash`, and write through the same handle, all under the lock. Closes the TOCTOU clobber window. Add `fd-lock` to `Cargo.toml`.
2. **Annotation-store save serialization** (`stores/annotations.ts:74-91`) — replace the unsequenced `await save()` in every mutator with a serialized chain (`saveChain = saveChain.then(runSave)`) so concurrent mutations (e.g. debounced notes + a delete) cannot interleave and drop each other. Each link snapshots the latest store state at write time.
3. **Debounce flush on unmount** (`EditorPane.svelte` `onDestroy`) — if a debounce timer is pending, synchronously push `view.state.doc.toString()` into `tabsStore` before clearing it, so closing a tab / `{#key}` remount within the debounce window never discards the last keystrokes.
4. **Sidecar schema policy** (`annotations.rs:283-291`, used at `199-208`) — distinguish *malformed JSON* from *valid JSON missing `schema_version`*. The latter defaults to version `0` and migrates in place (mirroring `settings.rs:84`), instead of being quarantined as corrupt. Parse to `serde_json::Value` first to tell the two apart.
5. **CRLF-aware frontmatter** (`frontmatter.rs:53-90, 112-123, 147-156`) — match the opening/closing `---` fence with `\r?\n`; compute real byte offsets (e.g. iterate `split_inclusive('\n')` so `line.len()` includes the terminator) so `body_start` never drifts on `\r\n` lines. Collapse the two reassembly routes (`Display` impl vs `reassemble`) into one and guarantee a single trailing newline before the closing fence. **Preserve the document's original line endings** — never normalize/rewrite them.

### Acceptance
- A save whose `expected_hash` no longer matches disk returns `HASH_MISMATCH` and does **not** write, even under a concurrent external write.
- Rapid interleaved mutations (type a general note while deleting an annotation) persist both; no dropped write.
- Closing a tab mid-typing preserves the last edit in the store.
- A valid sidecar with no `schema_version` loads + migrates (is **not** renamed to `.bak`).
- A CRLF document with frontmatter renders with the YAML stripped and the body intact; round-tripping it preserves CRLF.

### Tests
- `file_io` save-lock / concurrent-write test.
- `annotations` missing-`schema_version` → migrate (not quarantine) test.
- `frontmatter` CRLF parse + byte-offset + reassemble round-trip tests.
- Frontend: annotation-store serialization test (interleaved mutations both persist); EditorPane unmount-flush test.

## WS-3 · Cheap perf / stability wins

**Files:** `src/lib/render/markdown.ts`, `src/lib/fx/fluid.ts`, `vite.config.ts` (optional chunk config).

### Changes
1. **Curate highlight.js** (`markdown.ts:228`) — import `highlight.js/lib/core` and register a curated grammar set instead of the full ~190-language build: `javascript, typescript, rust, python, go, json, yaml, toml, bash, sql, xml(html), css, markdown, diff, dockerfile, c, cpp, java` (tune as needed). Target: that chunk drops from ~260KB+ gz to ~30–50KB.
2. **Trim Mermaid** (`markdown.ts:246`) — register only the diagram types Revenant supports so the transitive `katex`, `cytoscape`, `d3`, `roughjs` engines are not pulled. **Acceptance criterion:** those four packages no longer appear in the production build output. Hoist `mermaid.initialize()` to a module-level run-once after the dynamic import (it is currently called per render). Coordinate with the future KaTeX work (#7) so math is added intentionally, not via the Mermaid back-door.
3. **Fix the WebGL leak** (`fluid.ts:566-575` `dispose()`) — explicitly `deleteTexture` / `deleteFramebuffer` every FBO (velocity, dye, divergence, curl, pressure, coverage read+write), `docTex`, `docBlur` read+write; `deleteProgram` every program; `deleteVertexArray(quadVao)` + delete the quad buffer; **then** call `loseContext()` (which may be `null` on WebView2/ANGLE — so deletion must not depend on it).

### Acceptance
- Production build no longer bundles `katex`, `cytoscape`, `d3`, `roughjs`; the hljs chunk is a fraction of its prior size (verify in the `vite build` chunk report).
- Opening many documents in a session does not accumulate GPU memory / WebGL contexts (no "too many contexts" warnings); a headless smoke confirms the transition still renders and tears down.

## Workstream / file-ownership mapping (for `feature-implement` worktrees)

Per the repo's CLAUDE.md ownership, the three WS split cleanly for parallel worktrees, with one coordinated touch-point on `ipc.rs`:
- **WS-1** touches `ipc.rs` (WS-A surface), `reanchor.rs` + `annotations.rs` (WS-B), `EditorPane.svelte` (WS-C). The `ipc.rs` edits (save/load annotation bodies) are additive and do not change the frozen command/type surface.
- **WS-2** touches `file_io.rs` + `annotations.rs` + `frontmatter.rs` + `Cargo.toml` (WS-B/A), `stores/annotations.ts` + `EditorPane.svelte` (WS-C).
- **WS-3** touches `render/markdown.ts` + `fluid.ts` + `vite.config.ts` (WS-C/A).

WS-1 and WS-2 both edit `annotations.rs` and `EditorPane.svelte`, so they should be sequenced (or merged into one Rust-data worktree) rather than run fully parallel. `feature-research` should plan the exact split.

## Risks / notes
- **Frozen contract preserved.** WS-1 adds no IPC command or type field; only command *bodies* change. The contract test stays green.
- **Re-anchor persistence on load** writes the sidecar during `load_annotations` (to bake in the new hash + positions). Must reuse the existing gitignore-on-first-write path and not trigger a spurious conflict/watcher event (record the write as internal, like `save_file` does).
- **Mermaid trim** must be validated against the diagram types real users render; the acceptance is "the four heavy engines are gone," not "drop diagrams users need."
